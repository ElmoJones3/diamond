# Storage System

Diamond's storage is built around two ideas: content-addressable storage (CAS) for deduplication, and a human-navigable versioned directory structure for the MCP server to read from.

## On-Disk Layout

```
~/.config/diamond/
└── registry.json                    ← manifest of all tracked libraries/repos

~/.local/share/diamond/
├── store/                           ← content-addressable store (CAS)
│   └── ab/                          ←   first 2 hex chars of hash (sharding)
│       └── abcdef1234...            ←   full SHA256 hash = file name = content
└── storage/                         ← versioned views (human-navigable)
    └── msw/
        ├── 2.12.10/                 ←   pinned version directory
        │   ├── api/
        │   │   └── handlers.md     ←   hardlink → store/ab/abcdef...
        │   ├── search-index.json   ←   MiniSearch index (keyword)
        │   └── vector-index.json   ←   Transformers.js index (semantic)
        └── latest -> 2.12.10        ←   symlink to most recent version

~/.cache/diamond/
└── temp-abcdef...                   ← staging area for atomic CAS writes
```

All paths are XDG-compliant and can be overridden with `XDG_DATA_HOME`, `XDG_CONFIG_HOME`, and `XDG_CACHE_HOME`.

## Content-Addressable Store (`CasStore`)

The CAS is an append-only object store where the address of every file is its SHA256 hash. This provides **deduplication** (identical pages are stored once) and **integrity** (content can be verified by hashing).

## Versioned Storage (`StorageManager`)

`StorageManager` creates a "project view" of the CAS using hardlinks. A hardlink points to the same disk blocks as the CAS file, meaning reading from `storage/` costs zero extra disk space and zero copy time. Deleting a versioned directory doesn't affect the CAS; the actual data is only freed when all hardlinks to it are gone.

## Search Indices (`SearchService`)

Every library or repository indexed by Diamond gets two companion files in its storage directory:

1. **Keyword Index (`search-index.json`)**: A MiniSearch inverted index optimized for fast, exact technical lookups and prefix matching.
2. **Semantic Index (`vector-index.json`)**: A collection of 384-dimensional vectors (embeddings) generated by `all-MiniLM-L6-v2`.

These indices are built once at sync/add time and loaded on demand. For search, Diamond performs **hybrid reranking**: keyword matches provide high precision, while vector similarity ensures high recall for conceptual queries.

## Registry (`RegistryManager`)

The registry is a single JSON manifest that tracks the metadata for every library (versions, sync timestamps) and repository (local paths, auto-sync settings). It is the source of truth for the `list_registry` tool.
